# P090 – Feature Flag System for Safe Incremental Rollout

## Status
Not Started

## Dimensions
Reliability · Maintainability · Performance

## Problem

As SketchGit grows, new features need to be tested with a subset of users before full
rollout. Currently:

| Scenario | Risk without feature flags |
|----------|--------------------------|
| A new canvas tool is deployed | All users see it immediately; no gradual rollout |
| A new collaboration feature has a bug discovered post-deploy | Full rollback required (downtime risk) |
| A performance-critical change (e.g. P088 read replica routing) needs A/B testing | No mechanism to route a percentage of traffic to new code path |
| A large proposal (e.g. P085 canvas schema versioning) has multiple phases | All phases must be deployed atomically or left behind feature branches indefinitely |
| A beta feature needs to be enabled only for specific rooms or users | Hard-coded per-user logic pollutes business code |

No existing proposal addresses controlled, runtime-configurable feature gates.

## Proposed Solution

Implement a **lightweight, self-hosted feature flag system** that stores flags in
the existing PostgreSQL database. This avoids adding an external SaaS dependency and
keeps the flag state auditable via the existing Prisma migration and event log
infrastructure.

### Design principles

- **Simple**: flags are boolean values (on/off). No percentage rollout in v1.
- **Targeted**: a flag can be enabled globally, per-room, or per-user.
- **Fast**: flag reads are cached in-process (LRU cache, TTL 30 s) to avoid a DB
  round-trip on every request.
- **Observable**: flag evaluations are logged via Pino; flag changes are written as
  `RoomEvent` entries (P074) when room-scoped.
- **No external dependency**: implemented with Prisma + the existing LRU cache
  (`lib/cache/`).

### Database model

Add to `prisma/schema.prisma`:

```prisma
model FeatureFlag {
  id          String    @id @default(cuid())
  name        String    @unique
  description String    @default("")
  enabled     Boolean   @default(false)
  /// JSON: { userIds?: string[], roomIds?: string[] }
  targetScope Json      @default("{}")
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}
```

### Flag evaluation logic

`lib/server/featureFlags.ts`:

```typescript
async function isEnabled(
  flagName: string,
  context: { userId?: string; roomId?: string }
): Promise<boolean>
```

Evaluation order:
1. Check the in-process LRU cache (TTL: 30 s).
2. On cache miss, query `FeatureFlag` from the DB.
3. If `flag.enabled === false` and `targetScope` is empty → return `false`.
4. If `flag.enabled === true` → return `true` (globally on).
5. If `flag.enabled === false` but `targetScope.userIds` includes `context.userId` → return `true`.
6. If `flag.enabled === false` but `targetScope.roomIds` includes `context.roomId` → return `true`.
7. Default → return `false`.

### Admin API

Add `POST /api/admin/feature-flags` and `PATCH /api/admin/feature-flags/[name]`
routes, protected by a new `ADMIN` role on `RoomMembership` or a dedicated
`adminSecret` header checked against an `ADMIN_API_SECRET` environment variable.

These routes allow toggling flags at runtime without a deployment.

### Usage in application code

```typescript
// In an API route handler
const flagEnabled = await isEnabled('new-canvas-tool', { userId: session.userId });
if (!flagEnabled) return apiError(ApiErrorCode.NOT_FOUND, 'Not found', 404);

// In server.ts WebSocket handler
if (await isEnabled('presenter-mode', { roomId })) {
  // handle follow-request message
}
```

### Seed flags

Provide a `prisma/seed.ts` (or migration extension) that seeds the initial flags used
by upcoming proposals:

| Flag name | Default | Description |
|-----------|---------|-------------|
| `canvas-schema-v2` | `false` | Enable canvas JSON schema v2 writes (P085) |
| `read-replica` | `false` | Route reads to the replica (P088) |
| `sentry-client` | `false` | Enable Sentry browser SDK (P084) |
| `presenter-mode` | `true` | Enable presenter follow-view (P080) |

## Type Requirements

- `FeatureFlag` type is auto-generated by Prisma.
- `EvaluationContext` type: `{ userId?: string; roomId?: string }`.
- `targetScope` should be typed via a Zod schema to avoid raw JSON access:
  ```typescript
  const TargetScopeSchema = z.object({
    userIds: z.array(z.string().cuid()).optional(),
    roomIds: z.array(z.string().cuid()).optional(),
  });
  ```
- All flag names should be typed as a union of known flag name strings to prevent
  typo-based flag misses:
  ```typescript
  type FeatureFlagName =
    | 'canvas-schema-v2'
    | 'read-replica'
    | 'sentry-client'
    | 'presenter-mode';
  ```

## Linting Requirements

No new ESLint rules required. The `no-floating-promises` rule (P042) already forces
`await` on `isEnabled()` calls.

## Test Requirements

| Test | File |
|------|------|
| `isEnabled` returns `false` for unknown flag | `featureFlags.test.ts` |
| `isEnabled` returns `true` when `enabled: true` globally | `featureFlags.test.ts` |
| `isEnabled` returns `true` for targeted userId when globally off | `featureFlags.test.ts` |
| `isEnabled` returns `false` for non-targeted userId | `featureFlags.test.ts` |
| LRU cache is hit on second call (no second DB query) | `featureFlags.test.ts` |
| Admin POST creates a new flag | `route.test.ts` |
| Admin PATCH enables an existing flag | `route.test.ts` |
| Admin route returns 403 without admin secret | `route.test.ts` |

Use factory function `makeFeatureFlag()` in `lib/test/factories.ts` (P077 pattern).

## GitHub Copilot Agents and Skills

- A custom Copilot skill `toggle-feature-flag` allows agents to enable or disable a
  flag for a specific room or user during debugging: "Enable canvas-schema-v2 for
  room abc123".
- Copilot Chat can list available flags by querying `FeatureFlagName` type, ensuring
  it uses valid names in generated code.
- The Coding Agent should check whether a feature is gated by a flag before
  implementing a new code path and wrap it in `isEnabled()` where appropriate.
