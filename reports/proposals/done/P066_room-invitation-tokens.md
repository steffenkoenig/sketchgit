# P066 – Time-Limited Signed Room Invitation URLs

## Title
Replace Plain Room ID Sharing Links with HMAC-Signed, Time-Limited Invitation Tokens to Prevent Unauthorized Room Access

## Brief Summary
The current room sharing mechanism appends the plain room ID to the URL (e.g., `/?room=abc123`). Any user who obtains or guesses a room ID can join the room immediately, bypassing the `isPublic` flag for non-public rooms. Replacing or augmenting the sharing link with an HMAC-signed, time-limited invitation token—generated server-side, stored in a `RoomInvitation` table, and validated on WebSocket upgrade—closes this gap for private rooms while preserving frictionless access for public rooms.

## Current Situation
In `collaborationManager.ts`, the invite link is constructed as:
```typescript
private roomInviteLink(roomId: string): string {
  const url = new URL(window.location.href);
  url.searchParams.set('room', roomId);
  return url.toString();
}
```
The room ID is used directly as a query parameter. Anyone who knows the URL (e.g., obtained from a browser history, link preview, or URL prediction) can connect to the room's WebSocket endpoint and join as a peer.

In `server.ts`, the WebSocket upgrade handler (P034) enforces room access control:
```typescript
// P034: enforce access control on WS upgrade
const isPublic = await prisma.room.findUnique({ where: { id: roomId }, select: { isPublic: true } });
if (!isPublic?.isPublic && !isMember) {
  socket.destroy();
  return;
}
```
However, this only blocks non-members from **private** rooms. For public rooms, any authenticated (or anonymous, since P007 allows anonymous users) client can join. For private rooms, the only way to join is to already be a member—there is no mechanism to invite someone.

## Problem with Current Situation
1. **Private rooms cannot be shared**: There is no way to invite a non-member to a private room. The only option is to make the room public, which removes all access control.
2. **Room ID is a weak secret**: Room IDs are generated by the client as short random strings (16 hex characters from `crypto.randomUUID()`). While unpredictable in practice, they provide no time-limited access guarantee.
3. **No invitation audit trail**: There is no record of who shared a room link with whom, or when an invitation was used. This matters for compliance (P041 GDPR) and security auditing.
4. **No expiry on shared links**: A room link shared in a chat message or email remains valid forever. There is no way to revoke access without removing the user from room membership.
5. **Anonymous join for public rooms is unrestricted**: Any anonymous user can join any public room, including one that was originally shared only with a specific group. Signed tokens allow public rooms to require a valid invitation link for first-time access.

## Goal to Achieve
1. Add a `RoomInvitation` Prisma model with fields: `token` (HMAC-signed, opaque), `roomId`, `createdBy`, `expiresAt`, `maxUses`, `useCount`, `createdAt`.
2. Add `POST /api/rooms/[roomId]/invitations` (owner only) to create a new invitation token.
3. Add `GET /api/invitations/[token]` (public) to validate a token and redirect to the room (adding the user as a member if the room is private).
4. Update the WebSocket upgrade handler to accept either room membership or a valid, unused invitation token presented as a query parameter.
5. Update `collaborationManager.ts` to use the new invitation link format when inviting is enabled.

### 2. Create invitation token generator in `lib/server/invitationTokens.ts`
```typescript
import { createHmac, randomBytes, timingSafeEqual } from 'node:crypto';

const SECRET = process.env.INVITATION_SECRET ?? process.env.AUTH_SECRET!;

/**
 * Generate a cryptographically random invitation token.
 * The token is 32 random bytes encoded as hex (64 characters).
 * No HMAC is needed: the token is stored in the database and validated
 * by lookup rather than by signature, so the randomness alone provides
 * unguessability.
 */
export function generateInvitationToken(): string {
  return randomBytes(32).toString('hex');
}

/**
 * Generate a signed invitation URL.
 * The HMAC over (token + roomId + expiresAt) allows the server to verify
 * the URL parameters have not been tampered with, even before a database lookup.
 */
export function signInvitationToken(
  token: string,
  roomId: string,
  expiresAt: number,
): string {
  return createHmac('sha256', SECRET)
    .update(`${token}:${roomId}:${expiresAt}`)
    .digest('hex');
}

export function verifyInvitationSignature(
  token: string,
  roomId: string,
  expiresAt: number,
  signature: string,
): boolean {
  const expected = signInvitationToken(token, roomId, expiresAt);
  // Use crypto.timingSafeEqual() for constant-time comparison to prevent
  // timing attacks on the HMAC signature verification.
  if (expected.length !== signature.length) return false;
  return timingSafeEqual(Buffer.from(expected), Buffer.from(signature));
}
```

> **Note**: Import `timingSafeEqual` from `node:crypto` alongside `createHmac` and `randomBytes`.

## Additional Considerations

### Default expiry and use limits
Sensible defaults: `expiresInHours = 24` (24-hour links), `maxUses = 10` (to allow sharing in a team). The room owner can override both. Single-use links (`maxUses = 1`) are appropriate for sensitive private rooms.

### Public rooms
For public rooms, invitation tokens are optional (the room is already accessible by URL). The invitation system is primarily useful for private rooms. Optionally, public rooms can require a valid invitation token for first-time anonymous access.

### Invitation revocation
A room owner can revoke all outstanding invitations by calling `DELETE /api/rooms/[roomId]/invitations`, which deletes all `RoomInvitation` records for the room.

### GDPR considerations
`RoomInvitation` records should be included in the user data export (P041). Invitations created by a deleted user (`createdBy = null` after `onDelete: SetNull`) remain valid until their expiry time.

### Integration with the room pruning job (P032)
The room pruning job (P032) already deletes inactive rooms. When a room is deleted (via `onDelete: Cascade`), all associated `RoomInvitation` records are automatically deleted.

### Rate limiting
The `POST /api/rooms/[roomId]/invitations` endpoint must be covered by the rate limiter (P015/P046) to prevent invitation token flooding. Recommend a limit of 20 invitations per room per hour.

## Testing Requirements
- `generateInvitationToken()` returns a 64-character hex string.
- `verifyInvitationSignature()` returns `true` for a valid signature and `false` for a tampered token.
- `POST /api/rooms/[roomId]/invitations` returns 401 for unauthenticated requests.
- `POST /api/rooms/[roomId]/invitations` returns 403 for non-owner users.
- `POST /api/rooms/[roomId]/invitations` returns a `url` with the signed token.
- `GET /api/invitations/[token]` with an expired token returns 410 (Gone).
- `GET /api/invitations/[token]` with a fully-used token (useCount >= maxUses) returns 410.
- `GET /api/invitations/[token]` with a valid token increments `useCount` and redirects.
- WebSocket upgrade with a valid `invite` token grants access to a private room for a non-member.
- WebSocket upgrade with an expired or invalid token denies access.
